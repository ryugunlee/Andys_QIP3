# Andys_QIP3


## 프로그램 개요

이 프로젝트는
"투자자가 스스로 의사결정을 내릴 수 있도록 충분한 정보를 제공하는 것"
을 목표로 한다.
매수/매도를 강요하지 않는다.

모든 정보는 판단을 돕는 자료이다.
이 프로그램은 기본적으로 주식 매매에 있어 좋은 의사결정을 하는 것을 돕기 위해 만들어졌다.
좋은 의사결정을 돕기 위하여, 기업에 대해 정량적/정성적 평가를 실행하고, 주가가 저평가된 건전하고 좋은 기업들을 선별한다.

이 프로그램은 더 나아가, 개인들이 포트폴리오를 짜는 것을 도와줄 수 있다.


## 개발 철학

### 기존 코드 존중

현재 프로젝트의 구조를 최대한 존중한다.
새로운 기능을 구현하기 위해 기존 구조를 불필요하게 변경하지 않는다.
구조 개선이 필요할 경우 그 이유와 장단점을 먼저 설명하고 허가를 받은 후 진행한다.

### 기타 철학

모듈화, 캡슐화, 추상화, 계층화의 4가지 원칙을 지켜 복잡한 프로그램이지만 유지보수가 용이하고 단순하게 구상하고 개발한다.
보여지는 부분에서는 가독성을 반드시 우선하며, 유저가 프로그램이 제시하는 정보를 받아들일 때 편할 수 있는 최대한의 구조로 프론트 구조를 만든다.
확장의 여지가 있는 부분은, 폴더를 따로 만들어서 파일을 보관하고, 코드가 단일 파일 내에서 이유 없이 너무 길어지거나 복잡하게 만들지 않는다.
가능한 작은 단위에서 수정하여 최대한의 효과를 내도록 노력한다.
기존 기능을 변경하거나 삭제하지 않는다.
변경이 필요한 경우 반드시 영향 범위를 설명한다.


## 코드 스타일

함수는 가능한 하나의 책임만 가지게 만든다. 50줄 이하가 권장되나, 가독성과 책임 분리가 더 중요하다.
변수명은 의미가 있도록 한다.
TypeScript strict
Python type hint 사용
Magic Number 금지. 단 의미가 명확한 상수는 예외로 한다.


## 파일 작성 원칙

한 파일은 하나의 책임이다. 다른 파일과 너무 유기적으로 결합하지 않아야 하며, 구분 단위로서의 의미를 가질 수 있어야 한다.

## UI/UX 원칙

정보량이 화면에 많이 나오더라도, 화면이 답답하지 않아야 한다.
PC 환경을 중점으로 개발하지만, 모바일 환경도 충분히 고려해야 한다.
애니메이션보다 반응 속도를 우선 생각한다.
기능은 숨기지 말고 직관적으로 배치한다. 유저가 쉽게 이것이 무슨 기능인 지 감을 잡을 수 있게 만든다.

## AI 작업 방식

### 계획을 만들 때

현재 코드 구조를 먼저 분석한다.
기존 함수와 컴포넌트를 최대한 재사용한다.
같은 기능을 두 번 구현하지 않는다.
구현된 기존 기능이 있다면 그것을 활용한다. 커밋 내역과 .claude/STRUCTURE.md를 확인하여 찾는다.

### 계획을 제시할 때

계획을 제시할 때는 이하 순서에 맞게 계획을 제시한다.
1. 요구 사항을 어떻게 해석했는지 구체적으로 제시.
2. 요구 사항에 대하여 가능한 부분과 불가능한 부분을 체크하여 제시
3. 가능한 부분에 대하여 어떤 모습으로 진행되어 나타나는지 제시
4. 불가능한 부분에 대하여 불가능한 이유와 한계를 제시하고, 대신할 수 있는 선택지를 생각해서 제시
5. 구조 개선이 필요한 부분에 대한 간단한 설명과, 그 구조 개선과 작업이 가져오는 리스크와 리턴 설명. 큰 구조 개선일 경우 반드시 허가 받아야함.
6. 내용을 종합하면서, 대답을 받아야 하는 부분이나 구체화가 필요한 부분에 대하여 질문하면서, 코드 수정에 대하여 허가 받기.

프로젝트의 구조나 의도를 추측하지 않는다.
확실하지 않은 경우 질문한다. 비슷한 부류나 연계된 질문은 한 번에 질문한다.

### 계획을 실행할 때

계획을 실행할 때는 큰 범주에서의 구조 개선은 최소화하고, 구조 개선시 반드시 유지보수성을 고려할 것.
모듈화와 캡슐화에 집중하고 이 규칙을 지키기 위해 노력하기.
계획을 실행하면서 느낀 코드의 문제점이나 개선의 여지가 있는 사항은 따로 .claude/PROBLEMS.md에 저장하기
기능(함수)을 만들면 반드시 .claude/STRUCTURE.md 에 그 기능이 어디에 어떤 모습으로 있는지, 어떤 기능인지 저장한다.
