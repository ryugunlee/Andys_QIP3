# Andys_QIP3


## 프로그램 개요

이 프로젝트는
"투자자가 스스로 의사결정을 내릴 수 있도록 충분한 정보를 제공하는 것"
을 목표로 한다.
매수/매도를 강요하지 않는다.

모든 정보는 판단을 돕는 자료이다.
이 프로그램은 기본적으로 주식 매매에 있어 좋은 의사결정을 하는 것을 돕기 위해 만들어졌다.
좋은 의사결정을 돕기 위하여, 기업에 대해 정량적/정성적 평가를 실행하고, 주가가 저평가된 건전하고 좋은 기업들을 선별한다.

이 프로그램은 더 나아가, 개인들이 포트폴리오를 짜는 것을 도와줄 수 있다.

## 아키텍처
이 파일은 .claude의 STRUCTURE.md와는 다른 의미를 가진다. 함수보다는 구조의 영역이기 때문이다.

영역의 대분류는 3개이며, 진행의 순서에 따라 정렬하면 이렇다.
### 데이터 수집
데이터를 수집하는 영역이다. 질 좋은 데이터를, 가능한 한 많이 구하는 것이 목적이다.
추가로 국제유가, 기준금리 등 다양한 세계 경제적인 요인도 체크한다.
다양한 부분에서 데이터를 구한다. 신뢰도가 높은 곳에서 더 데이터를 많이 구한다.
일차적으로 기업들을 서로 비교할 수 있는 분석 지표를 만든다. (PER, PSR, PEGR, 부채비율 등)

### 데이터 분석
데이터를 분석하는 영역이다. 수집 영역에서 가져온 분석 지표와 수집된 데이터를 토대로 하여 각 기업들의 점수를 산출한다.
수익성, 가치성, 가격 변화 모멘텀, 안정성, 재무건전성, 미래가치성 등을 평가한다.
이후 좋은 주식을 선별하며, 선별 요건 및 방식은 QUANT.md에 저장된다.
선별된 주식들을 토대로 기업 뉴스 및 정보를 조사하고, 그에 따라 AI를 통해 정성적으로, 그리고 정치적, 사회적 관점에서 주식의 가치를 평가한다.
추가로 산업과 섹터에 대해서도 기업들 전체를 통하여 섹터와 산업 각각을 비교한다.

### 데이터 표현
분석된 데이터를 표현하는 영역이다.
다양한 그래프와 표를 통하여 시각적으로 표현하며, 가치투자와 퀀트투자에 대해 기본적인 지식이 있는 투자자를 위해 만들어졌다.
가능한 분석 및 수집을 완료한 모든 데이터를 표현할 수 있게 노력해야 한다. 적어도 볼 수 있게 해줄 수 있어야 한다.

### 데이터 수집
기업 및 주식의 데이터를 수집하는 부분이다.
KOSPI, KOSDAQ, NASDAQ, NYSE 등 한국/미국 주식시장에 상장한 기업들의 데이터를 분석한다.
웹 크롤링이나 API를 활용한다.
기업의 재무제표, 밸런스시트, 현금흐름표에 나와 있는 내용을 전부 수집한다.
수집한 정보를 토대로 PER, PBR, 이자보상비율, 연매출증가율 등 주식 분석에 필요한 지표들을 만든다.(간단한 비율 같은 경우는 재무제표 정보를 통해서 만들 수 있기 때문이다. 어떤 정보를 찾고 활용할 지에 대해서는 추후 여기에 구체화한다.)
분석 영역에서 추가 조사가 필요한 주식을 받으면, 추가 조사를 진행한다.(뉴스기사 등에 대한 분석 조사)
기업 뿐만 아니라, 금리, 10년물 채권 이자율, 유가흐름, 금 가격 흐름, 인플레이션, 진행 중인 전쟁 및 세계적 이슈, 환율(미달러 원 환율, 엔화 원 환율, 위안화 원 환율 등)을 수집한다.

### 데이터 분석
데이터 수집 영역에서 수집한 데이터를 분석하여 점수화하고, 의미 있는 기업을 분류 및 선발한다.
수집한 데이터를 토대로, 준 최고점과 최저점에서의 위치를 통하여 점수를 내는 standardscore, 퍼센티지 위치를 통해 내는 rankscore를 확인한다.
투자에 적정한 주식들 몇 개에 한하여 정성적 조사를 실행한다.(뉴스기사 확인하기, 산업 전망 및 산업에서의 위치 확인하기 등)
정성적 조사 이후 장래가 탄탄한 주식을 분류한다.

### 데이터 표현
데이터 수집, 분석 영역에서 분석한 데이터를 웹페이지, 앱, 메일 등으로 보내고 표현한다. 
페이지에서는 기업 외 국가, 금리 등 경제흐름에 관한 정보도 제시할 수 있어야 한다.
추천하는 기업을 기본 페이지에서 제시한다.
섹터/산업/기업을 검색할 수 있고, 산업이나 섹터를 검색한 경우 추천하는 기업을 가장 먼저 보여준다.
기업에 대하여 찾아 클릭하면 기업에 대한 정량/정성적 분석 정보를 볼 수 있다. (정성적 분석을 하지 않았을 경우 정성적 분석은 나오지 않는다.)
AndysNote를 연동하여 보고서를 볼 수도 있다.(이건 아직 개발 예정 아님)



## 기술 스택

현재 구현된 부분(데이터 수집/분석)은 Python 3 기반이다.
- 데이터 처리: pandas, numpy
- 데이터 수집: yfinance(미국 등 해외 종목), 네이버증권 API(한국 KRX/KOSPI/KOSDAQ, requests로 직접 호출),
  FinanceDataReader(티커 목록)
- 저장: DuckDB (일봉/재무제표/분석 스냅샷 — CSV/txt 산출물을 대체)
- 기타: tqdm(진행 표시), schedule(정기 실행)
- 의존성은 requirements.txt에 명시한다. 새 라이브러리를 쓰면 반드시 여기에 추가한다.

데이터 표현 영역(웹 프론트)은 아직 계획 단계이며, 개발 시 TypeScript를 사용할 예정이다.
(코드 스타일의 "TypeScript strict" 규칙은 이 미래 웹 개발에 적용된다.)

## 실행 방법

```bash
pip install -r requirements.txt
python Andys_QIP2.py   # 실행 후 시장명 입력 (예: NASDAQ, NYSE, KRX, AMERICAN)
```

- 결과물은 `qipinfos/andys_qip.duckdb`(DuckDB)에 저장된다. 이 폴더는 git에 커밋하지 않는다.
- 자동화된 테스트는 아직 없다.
- GitHub Actions로 요일별 시장 수집(월 KOSPI/화 KOSDAQ/수 NASDAQ/목 NYSE)과 매일 매크로 수집이
  자동 실행된다 (`.github/workflows/`). git에 커밋하지 않는 DuckDB는 저장소의 비공개(draft)
  릴리스(`data-store`)에 자산으로 올려 실행 간 영속시킨다 — 자세한 내용은
  `.claude/STRUCTURE.md`의 자동화 섹션 참고.

## 개발 철학

### 기존 코드 존중

현재 프로젝트의 구조를 최대한 존중한다.
새로운 기능을 구현하기 위해 기존 구조를 불필요하게 변경하지 않는다.
구조 개선이 필요할 경우 그 이유와 장단점을 먼저 설명하고 허가를 받은 후 진행한다.

### 기타 철학

모듈화, 캡슐화, 추상화, 계층화의 4가지 원칙을 지켜 복잡한 프로그램이지만 유지보수가 용이하고 단순하게 구상하고 개발한다.
보여지는 부분에서는 가독성을 반드시 우선하며, 유저가 프로그램이 제시하는 정보를 받아들일 때 편할 수 있는 최대한의 구조로 프론트 구조를 만든다.
확장의 여지가 있는 부분은, 폴더를 따로 만들어서 파일을 보관하고, 코드가 단일 파일 내에서 이유 없이 너무 길어지거나 복잡하게 만들지 않는다.
가능한 작은 단위에서 수정하여 최대한의 효과를 내도록 노력한다.
기존 기능을 가능한 변경하거나 삭제하지 않는다.
변경이 필요한 경우 반드시 영향 범위를 설명한다.
맨 처음 있던 Andys_QIP2.py의 코드를 무조건 따를 필요는 없다. 그러나, 그 이후 작성된 디테일과 부분들은 한 번 제대로 만들었으면 특별한 이유가 없이 수정하지 않는다.


## 코드 스타일

함수는 가능한 하나의 책임만 가지게 만든다. 50줄 이하가 권장되나, 가독성과 책임 분리가 더 중요하다.
변수명은 의미가 있도록 한다.

Python type hint 사용
Magic Number 금지. 단 의미가 명확한 상수는 예외로 한다.

한 파일은 하나의 책임이다. 다른 파일과 너무 유기적으로 결합하지 않아야 하며, 구분 단위로서의 의미를 가질 수 있어야 한다.

## UI/UX 원칙

정보량이 화면에 많이 나오더라도, 화면이 답답하지 않아야 한다.
PC 환경을 중점으로 개발하지만, 모바일 환경도 충분히 고려해야 한다.
애니메이션보다 반응 속도를 우선 생각한다.
기능은 숨기지 말고 직관적으로 배치한다. 유저가 쉽게 이것이 무슨 기능인 지 감을 잡을 수 있게 만든다.

## AI 작업 방식
반드시 유저가 읽는 문서를 작성할 때 한국어(한글)로 작성한다.
계획하거나 진행을 먼저 고려할 때가 있을 텐데, 그럴 때 궁금한 점이나, 확실하지 않은 점이 있으면 즉시 유저에게 질문한다.
맨 처음 있던 Andys_QIP2.py의 코드를 무조건 따를 필요는 없다. 그러나, 그 이후 작성된 디테일과 부분들은 한 번 제대로 만들었으면 특별한 이유가 없이 수정하지 않는다.
작업 중에 유저에게 질문하거나 하는 등 개발 방향과 결정사항에 대한 변화가 있을 경우 .claude/DECISIONS.md 에 기록하여 저장한다.

### 계획을 만들 때

현재 코드 구조를 먼저 분석한다.
기존 함수와 컴포넌트를 최대한 재사용한다.
같은 기능을 두 번 구현하지 않는다.
구현된 기존 기능이 있다면 그것을 활용한다. 커밋 내역과 .claude/STRUCTURE.md를 확인하여 찾는다.
막히는 부분이나 조금 께름칙한 부분이 있으면 .claude/PROBLEMS.md를 확인하여 참고한다.
이전 결정사항 등에 대해 궁금하거나 참고해야 한다면 .claude/DECISIONS.md를 참고한다. 만약 DECISIONS.md를 전부 참고했는데도 충돌하는 문제는 다시 질문한다.
docs폴더에 변경사항을 바로바로 대입할 수 있게 만들어라. 데이터 표현 영역을 건드릴 때는 반드시 docs에 대한 수정도 참고할 것. 

### 계획을 제시할 때

계획을 제시할 때는 이하 순서에 맞게 계획을 제시한다.
1. 요구 사항을 어떻게 해석했는지 구체적으로 제시.
2. 요구 사항에 대하여 가능한 부분과 불가능한 부분을 체크하여 제시
3. 가능한 부분에 대하여 어떤 모습으로 진행되어 나타나는지 제시
4. 불가능한 부분에 대하여 불가능한 이유와 한계를 제시하고, 대신할 수 있는 선택지를 생각해서 제시
5. 구조 개선이 필요한 부분에 대한 간단한 설명과, 그 구조 개선과 작업이 가져오는 리스크와 리턴 설명. 큰 구조 개선일 경우 반드시 허가 받아야함.
6. 내용을 종합하면서, 대답을 받아야 하는 부분이나 구체화가 필요한 부분에 대하여 질문하면서, 코드 수정에 대하여 허가 받기.

프로젝트의 구조나 의도를 추측하지 않는다.
확실하지 않은 경우 질문한다. 비슷한 부류나 연계된 질문 한 번에 질문한다.

### 계획을 실행할 때

계획을 실행할 때는 큰 범주에서의 구조 개선은 최소화하고, 구조 개선시 반드시 유지보수성을 고려할 것.
모듈화와 캡슐화에 집중하고 이 규칙을 지키기 위해 노력하기.
계획을 실행하면서 느낀 코드의 문제점이나 개선의 여지가 있는 사항은 따로 .claude/PROBLEMS.md에 저장하기
기능(함수)을 만들면 반드시 .claude/STRUCTURE.md 에 그 기능이 어디에 어떤 모습으로 있는지, 어떤 기능인지 저장한다.
또한 자동으로 한 단락의 계획을 오류 없이 실행하면 커밋 메시지를 요약해 적은 뒤 커밋한다!
데이터 표현 영역을 건드릴 때는 반드시 docs 파일의 웹 표현 워크플로도 고려한다.
